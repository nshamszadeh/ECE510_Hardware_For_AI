## 1. Dominant Kernel Identification Using Torchinfo

Profiling RAVE v2 inference (1 s of 48 kHz audio, CPU, 10 runs) with `torch.profiler` and
`torchinfo` across all four inference stages:

| Stage | Time/run | % time | Notes |
|---|---|---|---|
| PQMF filterbank | 0.80 ms | 1.5% | CachedPQMF analysis |
| **Variational encoder** | **25.67 ms** | **48.9%** | EncoderV2 strided + dilated Conv1d |
| Reparametrize | 0.26 ms | 0.5% | elementwise only |
| Decode (GeneratorV2 + inv. PQMF) | 25.80 ms | 49.1% | ConvTranspose1d + dilated Conv1d |

The encoder and decoder are approximately tied by timing. The **encoder (EncoderV2)** is selected
as the dominant hardware target because:
1. It is the higher-FLOPs stage when computed analytically (see §3 below).
2. Its output — a 128-dimensional latent vector at 24 Hz — defines the system interface: hardware
   accelerates the encoder and passes the compact latent to software for decoding.
3. The dominant low-level kernel, `aten::mkldnn_convolution`, accounts for **76.9% of total
   profiler CPU time** and originates primarily inside the encoder's strided Conv1d layers.

### 2.1 Why torchinfo Miscounts FLOPs for cached_conv
**How torchinfo works.** torchinfo computes FLOPs by registering forward hooks on every module
in the network. When a module's `forward()` is called, the hook fires, inspects the actual
input/output tensor shapes, and applies a FLOPs formula specific to the module type. For
`nn.Conv1d`, the formula is exact:

```
mult-adds = C_in × k × C_out × T_out
```

This works because torchinfo has an explicit handler registered for `torch.nn.Conv1d`.

**The problem: `cc.Conv1d` is not `nn.Conv1d`.** `cached_conv.Conv1d` (`cc.Conv1d`) is a
separate Python class from `torch.nn.Conv1d`. It is a streaming convolution wrapper that
maintains a causal history buffer for real-time inference — but from Python's perspective,
`isinstance(module, nn.Conv1d)` is `False` for a `cc.Conv1d` instance. When torchinfo's
dispatch logic finds no matching handler, it falls back to a heuristic based on parameter
tensors rather than the actual compute formula.

**How `weight_norm` makes it worse.** `EncoderV2` applies `weight_norm` to every Conv1d.
Weight normalization reparameterizes the weight tensor into two registered parameters:

```
weight_v:  shape (C_out, C_in, k)   — the weight direction vector
weight_g:  shape (C_out,)            — the per-filter scale
```

During forward, the actual convolution weight is recomputed on the fly as
`weight = weight_g × weight_v / ‖weight_v‖`. From torchinfo's perspective (when its fallback
heuristic runs), it sees two weight tensors instead of one, and the parameter count is:

```
params = C_out × C_in × k  +  C_out   (weight_v + weight_g)
```

The `weight_g` term is small, but the structural confusion is significant: the heuristic must
guess what the "weight" is without knowing how the two tensors combine.

**The resulting miscalculation.** The torchinfo fallback computes
`mult-adds ≈ total_params × C_out` instead of the correct
`mult-adds = C_in × k × C_out × T_out`. The ratio of erroneous to correct is approximately
`C_out / T_out`, which varies per layer and compounds across the module hierarchy, yielding the
observed ~8× aggregate overcounting. The exact fallback path is difficult to trace without
stepping through torchinfo's source, but the empirical pattern is consistent across every
`cc.Conv1d + weight_norm` layer in the encoder and decoder.

**Why PQMF is correct.** `CachedPQMF` also uses `cc.Conv1d`, but its filter coefficients
are stored as a plain buffer (no `weight_norm` reparameterization). torchinfo handles this
case correctly, producing the accurate 49.25 MFLOPs result that matches the analytical count.

| Module type | torchinfo handler | `weight_norm` | Result |
|---|---|---|---|
| `nn.Conv1d` (PQMF) | Exact match | No | Correct |
| `cc.Conv1d` + `weight_norm` (encoder/decoder) | No match → heuristic fallback | Yes (split tensors) | ~8× wrong |

### 3.1 Architecture Parameters (from `rave/configs/v2.gin` + `blocks.py`)

| Parameter | Value |
|---|---|
| N_BAND (PQMF output bands / encoder input channels) | 16 |
| CAPACITY | 96 |

| KERNEL_SIZE (dilated conv) | 3 |
| DILATIONS per ratio group | [1,3,9], [1,3,9], [1,3,9], [1,3] |
| LATENT_SIZE | 128 |
| n_out (encoder outputs mu and log_var) | 2 → output channels = 256 |

### 3.2 FLOP Formulas

**Conv1d (any strided or projection layer):**

$$\text{FLOPs} = 2 \times C_{\text{in}} \times k \times C_{\text{out}} \times T_{\text{out}}$$

where $k$ = kernel size (verified from parameter counts below), $T_{\text{out}}$ = output time
frames, and the factor of 2 accounts for one multiplication and one addition per MAC.

**DilatedUnit** (`Conv1d(C,C,k=3,dilation=d)` followed by `Conv1d(C,C,k=1)`; dilation does not
change FLOP count, only receptive field):

$$\text{FLOPs}_{\text{DilatedUnit}} = 2 \cdot C^2 \cdot 3 \cdot T + 2 \cdot C^2 \cdot 1 \cdot T = 8 C^2 T$$

### 3.3 Kernel Size Derivation

EncoderV2 uses `weight_norm`, which stores two tensors per Conv1d:
- `weight_v`: shape $(C_{\text{out}}, C_{\text{in}}, k)$ — the weight direction vector
- `weight_g`: shape $(C_{\text{out}},)$ — the per-filter scale

So `params = C_out × C_in × k + C_out`, giving `k = (params − C_out) / (C_in × C_out)`.

| Layer | C_in | C_out | params | Derived k | Cross-check |
|---|---|---|---|---|---|
| Initial Conv1d | 16 | 96 | 10,848 | (10848−96)/(16×96) = **7** | = 2×KERNEL_SIZE+1 ✓ |
| Strided conv, r=4 | 96 | 192 | 147,648 | (147648−192)/(96×192) = **8** | = 2×r ✓ |
| Strided conv, r=4 | 192 | 384 | 590,208 | **8** | ✓ |
| Strided conv, r=4 | 384 | 768 | 2,360,064 | **8** | ✓ |
| Strided conv, r=2 | 768 | 1536 | 4,720,128 | **4** | = 2×r ✓ |
| Final proj Conv1d | 1536 | 256 | 1,179,904 | **3** | = KERNEL_SIZE ✓ |

The strided downsampler kernel size is **k = 2r** (confirmed in `blocks.py` line 572).

| Parameter | Value |

| RATIOS (reversed for upsampling) | [2, 4, 4, 4] |
| KERNEL_SIZE (dilated conv) | 3 |

| `num_channels` (initial decoder width) | 2⁴ × 96 = 1,536 |

GeneratorV2 uses `cc.ConvTranspose1d` for upsampling (one per ratio stage) and the same
`cc.Conv1d` + `DilatedUnit` residual stacks as the encoder. The `normalization()` wrapper
applies `weight_norm` to every convolutional layer — so the same torchinfo miscounting applies
here as in the encoder (§2.1).

### 5.2 FLOP Formulas

**ConvTranspose1d** (stride = r, kernel = 2r, input length $T_{\text{in}}$):

Each of the $T_{\text{in}} \times C_{\text{in}}$ input elements fans out to $k \times C_{\text{out}}$ weight products:

$$\text{FLOPs}_{\text{ConvTranspose1d}} = 2 \times C_{\text{in}} \times k \times C_{\text{out}} \times T_{\text{in}}$$

**DilatedUnit**: same formula as encoder — $8 C^2 T$ evaluated at the post-upsample length.

### 5.3 Kernel Size Derivation

For `cc.ConvTranspose1d` with `weight_norm`, the weight direction tensor has shape
$(C_{\text{in}}, C_{\text{out}}, k)$ and the scale tensor has shape $(C_{\text{in}},)$, so
`params = C_in × C_out × k + C_in`, giving `k = (params − C_in) / (C_in × C_out)`.

| Layer | C_in | C_out | params | Derived k | Cross-check |
|---|---|---|---|---|---|
| Initial Conv1d | 128 | 1,536 | 591,360 | (591360−1536)/(128×1536) = **3** | = KERNEL_SIZE ✓ |
| ConvTranspose r=2 | 1,536 | 768 | 4,720,128 | (4720128−1536)/(1536×768) = **4** | = 2×r ✓ |
| ConvTranspose r=4 | 768 | 384 | 2,360,064 | (2360064−768)/(768×384) = **8** | = 2×r ✓ |
| ConvTranspose r=4 | 384 | 192 | 590,208 | (590208−384)/(384×192) = **8** | ✓ |
| ConvTranspose r=4 | 192 | 96 | 147,648 | (147648−192)/(192×96) = **8** | ✓ |
| Final Conv1d | 96 | 32 | 21,536 | (21536−32)/(96×32) = **7** | = 2×KERNEL_SIZE+1 ✓ |

The final Conv1d uses `kernel_size * 2 + 1 = 7` (confirmed in `blocks.py` line 679).
`C_out = 32` because `amplitude_modulation = True` doubles the output to `data_size × 2 = 32`

## 6. Interpretation and Final Dominant Kernel Identification

With AI ≈ 54 FLOP/byte, the encoder sits in the **memory-bandwidth-bound** regime on conventional
hardware (CPU/GPU ridge points are typically 100–1000 FLOP/byte). This is a direct consequence of
the weight-dominant byte traffic: the 61.3 MB of parameters must be streamed from DRAM for every
1-second inference frame, while the compute is only 3.45 GFLOPs.
**Note that the calculated roofline and kernel placements for our CPU shows the kernel is compute-bound rather than memory-bound; however this roofline plot does not take into account overflow of the encoder weight set out of the L3 cache. More detailed explanation for this discrepency is in partition_rationale.md**.

A custom hardware accelerator improves this by:
1. **On-chip SRAM weight storage** — weights are loaded once at startup and reused across frames,
   eliminating repeated DRAM access and dramatically increasing effective AI.
2. **Pipelined MAC arrays** — the regular, data-parallel structure of Conv1d maps directly to
   systolic arrays or parallel multiply-accumulate units.

These factors together move the design point above the ridge point, making it compute-bound
rather than memory-bound — which is where custom silicon achieves maximum efficiency gains over
a general-purpose processor.
